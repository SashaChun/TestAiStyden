# Feature Specification: UniPortal Student Cabinet

**Feature Branch**: `001-student-cabinet-frontend`  
**Created**: 2025-11-08  
**Status**: Draft  
**Input**: User description: "Фронтенд для кабінету студента з використанням React і TypeScript, дизайн з Figma"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Перегляд основної інформації та статистики (Priority: P1)

Студент відкриває кабінет і одразу бачить ключову інформацію про свою успішність: середній бал, кількість зароблених кредитів, пропуски та заборгованості. Ця інформація представлена у вигляді чотирьох статистичних карток з іконками та кольоровим кодуванням.

**Why this priority**: Це найважливіша функція, яка дає студенту миттєвий огляд його академічного статусу. Без цієї інформації кабінет втрачає основну цінність.

**Independent Test**: Можна протестувати завантаженням статичних даних студента та перевіркою відображення всіх чотирьох карток з коректними значеннями та стилізацією згідно Figma дизайну.

**Acceptance Scenarios**:

1. **Given** студент авторизований в системі, **When** відкриває головну сторінку кабінету, **Then** бачить 4 статистичні картки (середній бал, кредити, пропуски, заборгованості)
2. **Given** студент має середній бал 4.8, **When** переглядає картку "Середній бал", **Then** бачить значення "4.8" з зеленою іконкою
3. **Given** студент має 48 з 60 кредитів, **When** переглядає картку "Кредити", **Then** бачить значення "48/60" з синьою іконкою
4. **Given** студент має 2 пропуски, **When** переглядає картку "Пропуски", **Then** бачить значення "2" з помаранчевою іконкою
5. **Given** студент не має заборгованостей, **When** переглядає картку "Заборгованості", **Then** бачить значення "0" з фіолетовою іконкою

---

### User Story 2 - Перегляд розкладу на поточний день (Priority: P1)

Студент бачить розклад занять на сьогодні з деталями: назва предмету, викладач, аудиторія, час проведення та формат (онлайн/офлайн). Для онлайн занять доступна кнопка підключення до Zoom.

**Why this priority**: Розклад на сьогодні - критична інформація для щоденного використання. Студент повинен швидко знайти, куди йти і коли.

**Independent Test**: Можна протестувати з масивом занять на поточний день, перевіривши відображення всіх деталей кожного заняття та функціональність кнопки Zoom для онлайн занять.

**Acceptance Scenarios**:

1. **Given** сьогодні є 3 заняття, **When** студент переглядає розділ "Розклад на сьогодні", **Then** бачить список з 3 занять
2. **Given** заняття "Програмування" о 09:00-10:30 онлайн, **When** студент переглядає це заняття, **Then** бачить назву, викладача "Проф. Петренко А.В.", аудиторію "305", час та кнопку "Zoom"
3. **Given** заняття "Математичний аналіз" о 10:45-12:15 очно, **When** студент переглядає це заняття, **Then** бачить мітку "Очно" замість кнопки Zoom
4. **Given** студент клікає на кнопку "Переглянути все", **When** кнопка натиснута, **Then** відкривається повний розклад на тиждень/місяць

---

### User Story 3 - Навігація по розділах кабінету (Priority: P1)

Студент використовує бічну панель навігації для переходу між різними розділами: Головна, Розклад, Оцінки, Залікова книжка, Повідомлення, Профіль, Налаштування. Активний розділ виділяється візуально.

**Why this priority**: Навігація - основа UX. Без неї студент не зможе користуватися іншими функціями кабінету.

**Independent Test**: Можна протестувати клікаючи на кожен пункт меню та перевіряючи візуальну зміну активного стану та перехід на відповідну сторінку.

**Acceptance Scenarios**:

1. **Given** студент на головній сторінці, **When** переглядає бічну панель, **Then** пункт "Головна" виділений як активний
2. **Given** студент клікає на "Розклад", **When** клік виконано, **Then** відкривається сторінка розкладу і пункт "Розклад" стає активним
3. **Given** студент переглядає розділ "Повідомлення", **When** є 3 непрочитані повідомлення, **Then** біля пункту "Повідомлення" відображається бейдж з числом "3"
4. **Given** студент клікає на "Вийти", **When** клік виконано, **Then** відбувається вихід з системи

---

### User Story 4 - Перегляд непрочитаних повідомлень (Priority: P2)

Студент бачить список останніх повідомлень на головній сторінці з індикацією непрочитаних. Повідомлення включають тему, відправника та час відправки.

**Why this priority**: Важлива функція для комунікації, але не критична для базового функціоналу кабінету.

**Independent Test**: Можна протестувати з масивом повідомлень, де частина позначена як непрочитані, та перевірити візуальне виділення.

**Acceptance Scenarios**:

1. **Given** студент має 3 непрочитані повідомлення, **When** переглядає розділ "Повідомлення" на головній, **Then** бачить бейдж "3 нові"
2. **Given** є повідомлення "Зміна в розкладі", **When** студент переглядає список, **Then** бачить тему, час та індикатор непрочитаного
3. **Given** студент клікає на повідомлення, **When** клік виконано, **Then** повідомлення відкривається та позначається як прочитане

---

### User Story 5 - Перегляд поточних оцінок (Priority: P2)

Студент бачить таблицю з поточними оцінками по всіх предметах, включаючи назву предмету, викладача та оцінку у вигляді кругового індикатора.

**Why this priority**: Важлива інформація для відстеження успішності, але менш критична ніж загальна статистика.

**Independent Test**: Можна протестувати з масивом предметів та оцінок, перевіривши відображення таблиці та кругових індикаторів.

**Acceptance Scenarios**:

1. **Given** студент вивчає 5 предметів, **When** переглядає розділ "Поточні оцінки", **Then** бачить таблицю з 5 рядками
2. **Given** оцінка з "Програмування" - 92, **When** студент переглядає цей рядок, **Then** бачить круговий індикатор з числом "92"
3. **Given** студент переглядає таблицю, **When** наводить на рядок, **Then** рядок підсвічується для кращої читабельності

---

### User Story 6 - Перегляд профілю студента (Priority: P3)

Студент бачить свою особисту інформацію в шапці бічної панелі: фото, ім'я, група, курс та факультет.

**Why this priority**: Корисна інформація для ідентифікації, але не критична для основного функціоналу.

**Independent Test**: Можна протестувати з об'єктом даних студента, перевіривши відображення всіх полів.

**Acceptance Scenarios**:

1. **Given** студент авторизований, **When** переглядає бічну панель, **Then** бачить своє фото, ім'я "Марія Коваленко", групу "ІТ-21, 3 курс" та факультет "Факультет інформатики"
2. **Given** студент клікає на область профілю, **When** клік виконано, **Then** відкривається повна сторінка профілю

---

### Edge Cases

- Що відбувається, коли у студента немає занять на сьогодні? → Показується повідомлення "Сьогодні занять немає"
- Як система обробляє відсутність фото профілю? → Показується аватар за замовчуванням з ініціалами
- Що відбувається при помилці завантаження даних? → Показується повідомлення про помилку з кнопкою "Спробувати знову"
- Як відображаються дуже довгі назви предметів? → Текст обрізається з ellipsis (...)
- Що відбувається, коли посилання Zoom недоступне? → Кнопка Zoom неактивна або показує помилку
- Як система обробляє відсутність інтернет з'єднання? → Показується індикатор офлайн режиму
- Що відбувається при зміні розміру екрану? → Інтерфейс адаптується (responsive design)

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST відображати чотири статистичні картки: середній бал, кредити, пропуски, заборгованості
- **FR-002**: Система MUST відображати розклад занять на поточний день з деталями (предмет, викладач, аудиторія, час, формат)
- **FR-003**: Система MUST надавати кнопку підключення до Zoom для онлайн занять
- **FR-004**: Система MUST відображати бічну панель навігації з 7 пунктами меню
- **FR-005**: Система MUST виділяти активний пункт меню візуально
- **FR-006**: Система MUST відображати бейдж з кількістю непрочитаних повідомлень
- **FR-007**: Система MUST відображати профіль студента (фото, ім'я, група, курс, факультет)
- **FR-008**: Система MUST відображати список непрочитаних повідомлень на головній сторінці
- **FR-009**: Система MUST відображати таблицю поточних оцінок з круговими індикаторами
- **FR-010**: Система MUST використовувати дизайн-систему з Figma (кольори, шрифти, відступи, тіні)
- **FR-011**: Система MUST бути responsive та адаптуватися до різних розмірів екрану
- **FR-012**: Система MUST відображати стан завантаження даних (loading states)
- **FR-013**: Система MUST відображати помилки завантаження даних з можливістю повторної спроби
- **FR-014**: Користувачі MUST мати можливість вийти з системи через кнопку "Вийти"
- **FR-015**: Система MUST відображати іконки для кожного пункту меню та статистичної картки

### Key Entities

- **Student**: Представляє студента з атрибутами (id, ім'я, фото, група, курс, факультет, середній бал, кредити, пропуски, заборгованості)
- **ScheduleItem**: Представляє заняття з атрибутами (id, назва предмету, викладач, аудиторія, час початку, час кінця, формат, посилання Zoom)
- **Message**: Представляє повідомлення з атрибутами (id, тема, відправник, час, прочитане/непрочитане, текст)
- **Grade**: Представляє оцінку з атрибутами (id, предмет, викладач, оцінка, максимальна оцінка)
- **NavigationItem**: Представляє пункт меню з атрибутами (id, назва, іконка, шлях, бейдж)

### Non-Functional Requirements

- **NFR-001**: Інтерфейс MUST бути pixel-perfect відповідно до Figma дизайну
- **NFR-002**: Код MUST бути написаний на TypeScript з strict mode
- **NFR-003**: Компоненти MUST бути розділені на UI та логіку (separation of concerns)
- **NFR-004**: Компоненти MUST бути reusable та composable
- **NFR-005**: Код MUST відповідати принципам з constitution.md
- **NFR-006**: Час завантаження початкової сторінки MUST бути менше 2 секунд
- **NFR-007**: Інтерфейс MUST бути доступним (WCAG 2.1 AA)
- **NFR-008**: Код MUST бути покритий ESLint правилами без warnings

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Студент може побачити всю ключову інформацію (статистика + розклад) на головній сторінці за 1 клік
- **SC-002**: Інтерфейс візуально ідентичний Figma дизайну (pixel-perfect) на 95%+
- **SC-003**: Всі компоненти мають TypeScript типи без використання `any`
- **SC-004**: Код структурований згідно atomic design principles (atoms, molecules, organisms)
- **SC-005**: Навігація між розділами працює плавно без затримок
- **SC-006**: Responsive дизайн коректно працює на екранах від 320px до 1920px
- **SC-007**: Студент може підключитися до онлайн заняття за 2 кліки (відкрити кабінет → клік на Zoom)
- **SC-008**: Всі інтерактивні елементи мають візуальний feedback (hover, active states)
- **SC-009**: Помилки завантаження даних обробляються gracefully з зрозумілими повідомленнями
- **SC-010**: Код проходить type-check без помилок (`tsc --noEmit`)

## Technical Constraints

- **TC-001**: Використовувати React 18+ з функціональними компонентами та hooks
- **TC-002**: Використовувати TypeScript 5+ з strict mode
- **TC-003**: НЕ використовувати Tailwind CSS (використовувати CSS Modules або Styled Components)
- **TC-004**: Використовувати Vite як build tool
- **TC-005**: Слідувати структурі папок з constitution.md
- **TC-006**: Всі іконки мають бути SVG компонентами
- **TC-007**: Кольори, шрифти, відступи мають бути винесені в design tokens

## Out of Scope

- Backend API інтеграція (використовуються mock дані)
- Автентифікація та авторизація (припускається, що користувач вже авторизований)
- Редагування профілю студента
- Відправка повідомлень
- Детальний перегляд оцінок з історією
- Експорт даних
- Мобільний додаток (тільки responsive web)
- Темна тема (тільки світла)
- Мультимовність (тільки українська)
